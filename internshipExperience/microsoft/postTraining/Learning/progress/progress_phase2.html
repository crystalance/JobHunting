2. Harness hardening round 2 — speed + "always have output"

Two blockers made iteration painful: tasks took 15–21 min, and heavy runs produced no gradeable output. Fixes:

ProblemFixResult
Search too slow (3 workers, ~28s each)SEARCH_MAX_WORKERS 3 → 8~4× faster; 0 rate-limit issues
Over-search bloats history → context 400 → no outputsliding-window tool-history trim (_trim_tool_history, 90k-char budget) + safe_chat overflow trim-and-retrycontext never overflows
Over-search runs forever → 900s timeout → no outputhard search budget at the tool level (SEARCH_EPISODE_BUDGET): once N total searches hit → SEARCH_BUDGET_EXCEEDED → "write & stop"timeouts essentially eliminated

ws_en_073: 18-min ERROR-no-output → 4m50s OK-with-output. The hard budget matters because the prompt-level "~60 search" request is routinely ignored by a weak policy — it must be enforced in code.

3. Prompt fixes — target the three failure modes

Surgical edits to the base system prompt:

  • Anti-fabrication: "if not found after a retry, write nan — never invent, guess, or repeat a value across rows" (a fabricated value scores worse than an honest nan).
  • Write early & always have output: create the output file in the first few turns; never finish with no file.
  • No loop: if the same code fails twice, change approach.
  • Batch size 200 → 20 + a "~60 search budget" line (the runaway fan-out came straight from the old prompt).

Val with fixed prompt: F1_item 0.196 / F1_row 0.108 vs baseline 0.088 / 0.021 — ~2× F1_item, ~5× F1_row. Real but modest; confirms the data (not just the prompt) is the lever.

4. Rejection-sampling data pipeline (ReST / RFT)

Goal restated: SFT should teach the code-act pattern + crash-free code + always a gradeable output — not maximize F1 (that's RL's job). So we generate many teacher rollouts and keep the behaviorally clean ones.

gpt-5.4 × k per prompt→ verifier + deterministic gates→ LLM trajectory judge (§4f)→ retrain LoRA→ re-eval val

4.1 What we built

  • k-sampling — batch_eval --num_samples k --temperature (temperature only forwarded when set, so the gpt-5.4 reasoning model isn't broken; diversity comes from its inherent stochasticity). Each rollout → {iid}__t{n}. Validated: two samples of one instance genuinely diverged (one converged, one didn't).
  • Hard search budget (above) — bounds work per episode.
  • Trajectory scorer — post_training/sft/score_trajectories.py: per rollout, F1 (WideSearch verifier) + behavioral stats (turns, searches, error-rate, loop-hash, has-output, converged, constant-fill).

4.3 Selection — verifier + our own LLM judge (no third-party judge needed)

Filter is hybrid and mostly deterministic because we have gold labels (verifier-based, more reliable than LLM-as-judge):

  • Hard gates (code): output produced, no crash/timeout, fits context, valid tool-calls.
  • Code-quality gates (code): low error-rate, no loop, converged, bounded turns/searches.
  • Correctness floor (verifier): a low F1_item bar for anti-fabrication (you can't fabricate to it) — not a high F1_row bar, which would contradict the behavioral goal.
  • LLM trajectory judge (§4f, our own): an offline gpt-4.1 judge as a final top layer, scoring imitation-worthiness (clean code-act, sound reasoning, honesty, convergence). Ablatable; A/B'd before trusting.

Note: the constant_fill fabrication heuristic flagged 28% of the strong teacher's rollouts (false positives on legitimately repeated values), so it is a soft signal only — real fabrication detection is delegated to the LLM judge. Design: sft_data_generation_rejection_sampling.md.

5. Scoring, selection & the v2 retrain — the payoff

5.1 Scored all 280 rollouts

Behavioral quality is excellent and F1 is far above the old set:

  • Behavioral: 280/280 have output, 276/280 converged, 0 loops, mean error-rate 0.012.
  • F1_row mean 0.41 / median 0.42 / max 1.00  ·  F1_item mean 0.71 / median 0.82 / max 1.00.
  • 66/70 instances have ≥1 clean, f1_item≥0.3 sample — great coverage.

5.2 Selected 122 trajectories (trial-aware, top-2/prompt)

Filter: clean gates (output + converged + err≤0.2 + no-loop) + f1_item≥0.3, keep top-2 per prompt (ranked correctness-then-efficiency). 122 kept (vs the old 48) from 66 instances; 9 dropped (too long), 32 gated out. The LLM judge was held as an ablation — the data was already clean enough (err 0.012) that it would filter almost nothing.
