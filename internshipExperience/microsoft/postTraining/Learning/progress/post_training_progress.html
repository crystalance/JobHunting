1. The Task & Why It's Hard

WideSearch (ByteDance-Seed) gives an instruction that requires gathering many facts (rows × columns) from the live web and emitting a structured table. Scoring:

  • F1_item — fraction of individual cells correct (partial credit).
  • F1_row — fraction of rows whose every cell is correct.
  • score — strict all-or-nothing gate (usually 0 unless the whole table is perfect).

The agent runs a code-act loop: the policy LLM emits an execute_python tool call → code runs in a persistent IPython kernel → the kernel has a search_client (Bing grounding + an LLM that summarizes results) → the model reads stdout, iterates, and finally writes output/*.csv|md.

Policy LLM→ tool_call → IPython kernel→ search_client.search()→ Bing + LLM summary → stdout back to LLM↺ iterate → output CSV→ WideSearch eval

Difficulty for a small model: it must (a) reliably emit the tool-call protocol, (b) write syntactically valid Python every turn, (c) drive a long multi-turn loop without giving up, and (d) know when to stop and write the file. The base 7B fails on (b)–(d).

Why Qwen2.5-7B-Instruct (and why most academic RL papers use it)

  • De-facto research standard — the reference base for most 2024–2025 RL-for-reasoning/agents work (stable, reproducible, heavily documented).
  • Strong instruct + tool priors — follows the function-calling protocol out of the box; the code prior of the Coder variant actively fights the tool protocol.
  • Fully supported by our stack — vLLM 0.8.2 serving, verl/peft training, LoRA + full-FT all work; Qwen3 does not on this stack.
  • Right size — fits one A100-80GB for serving and leaves room for 4-GPU training.

Hardware: 4× A100-80GB. Serving env new_verl (vLLM 0.8.2, verl 0.2.0.dev, torch 2.6, peft 0.15.2, transformers 4.51.3). Harness runs in an isolated .venv-harness (py3.10).

3. Harness Engineering (prerequisite for any training)

Before training could produce signal, the harness had to reliably run an open model. Key fixes:

  • Tool-call recovery — small models emit structurally perfect tool calls but ~40% use Python single-quotes (invalid JSON). Added a repair ladder json.loads → brace-balance → ast.literal_eval → schema regex plus recovery of content-embedded <tool_call> blocks. Turned frequent premature stops into successful turns.
  • In-kernel search client bug — _bootstrap_kernel hard-coded an Azure client, so every in-kernel search_client.search() hit a bogus deployment and silently failed. Fixed to inject the configured (local/Azure) client.
  • Azure auth — DefaultAzureCredential(exclude_managed_identity_credential=True) (the VM's managed identity lacks OpenAI access; only the az-CLI user works).
  • Context — raised vLLM --max-model-len to 32768 (turn-4 overflow at 24576).
  • Crash hardening (2026-07-07) — search-heavy runs were killing the harness by an uncatchable OS signal. Added a per-cell search cap, a kernel-death guard + bounded stdout, and per-instance process isolation. Details: bug&fix/2026-07-07_silent_harness_crash.md.

Task: worldwide + New York / Detroit / Seattle restaurant counts for 5 burger brands at end-2024. Only 4 turns, source-backed, correctly leaves undisclosed city counts as nan.

T4 · write output & stop

Signature of the teacher: few turns, batched but purposeful searches, source-scoped queries (site:), and — crucially — it decides undisclosed cells are nan and stops. This "gather → verify → write → stop" arc is exactly what we want the student to imitate.

5.1 On over-searching & why we did NOT filter it out

The teacher searches heavily — across the 69 trajectories: median 58, mean 64, max 257 searches; 46/69 did ≥40. But this is batched wide fan-out (10–18 searches per cell), which is intrinsic to WideSearch (gather many rows), and all 48 kept trajectories still end by writing output. Filtering by search count would delete ~2/3 of the data and bias toward trivial tasks. Decision: keep the 48, cap searches at inference, and let RL trim efficiency later.

8. Decision & Next Steps

Full-parameter SFT vs GRPO — recommendation: go to GRPO

  • LoRA already achieved SFT's job — get the policy to complete the task and earn non-zero reward, so RL has a gradient. Done (0.80 F1_item, writes output).
  • Full-FT on 48 examples is risky — high capacity + tiny data ⇒ memorization / catastrophic forgetting. LoRA's low-rank constraint is a regularizer, which is why it's the better SFT choice at this data size.
  • The fixed benchmark hurts SFT, not RL — GRPO generates its own on-policy rollouts and optimizes F1 directly, so it is compute-limited, not data-limited.
  • Fallback if RL bootstrapping stalls — rejection-sampling fine-tuning (ReST/RFT): sample k trajectories per prompt from the strong teacher, keep those passing a reward bar (68 prompts × ~3 ≈ 200 trajectories) — more data without new prompts.

Immediate plan

  1. Finish the held-out val run → stable F1_row / F1_item / completion rate.
  2. If the cold-start holds, wire GRPO (verl) with the WideSearch F1 reward over the multi-turn agentic rollout, initialized from the LoRA SFT checkpoint.
  3. Add a turn/search cost term to the reward so RL trims the residual over-searching and the code-robustness loop.
  4. Final report on the untouched test split (086–100).
