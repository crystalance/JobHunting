2. v1 — original (stable, but limited)

R = 1.0·f1_item + 0.5·f1_row + 0.1·has_table − 0.1·turn_cost − 0.1·constant_fill
    turn_cost = min(n_turns/20, 1)      # INERT: n_turns never plumbed -> always 0

Behaviour: stable for 30 steps. The turn-cost term was dead (verl never passed num_turns), so efficiency was never explicitly rewarded — yet turns still dropped ~48% as an emergent effect of chasing outcome reward. Held-out f1 did not improve (bounded by coverage/search).

Lesson: this reward is safe but under-powered for coverage; the row term (0.5) is too weak and the efficiency term does nothing.

3. v2 — the failure (collapsed)

R = 1.0·f1_item + 0.8·f1_row + 0.05·has_table − 0.1·turn_cost − 0.1·constant_fill
    turn_cost = min(n_turns/12, 1)      # ACTIVE: n_turns = count("<tool_call>")
    KL 0.001 -> 0.01

What we changed: up-weighted coverage (0.5→0.8), shrank has_table, and activated the turn-cost (plumbed real turns, tighter budget) to make efficiency deliberate.

What happened: val f1 rose at step 5 (0.24→0.32) then collapsed — by step 15 the policy wrote no table at all (has_table→0), emitting one search tool-call and stopping.

Root cause — the turn-cost created a perverse "stop early" incentive. GRPO minimized the turn penalty by doing one search and terminating. That produces no table (reward ≈ 0), but once every rollout in a batch collapsed to no-table, all rewards were ~equal → advantage ≈ 0 → no gradient to recover. A reward-hacking → entropy/length-collapse absorbing state. Making it worse: keep-last-2 checkpointing deleted the one good (step-5) checkpoint.

4. v3 — new design proposed

R = 1.0·f1_item + 0.8·f1_row + 0.1·has_table − 0.1·constant_fill
    # NO turn-cost term.
    KL = 0.005 ;  init = SFT-v3 ;  best-by-val checkpointing

Design principles (each ties to a lesson)

DecisionWhy
Remove the turn-cost term entirelyIt caused the collapse. Efficiency must NOT fight task completion. The −48% turn win in v1 was emergent & safe — do not force it via a penalty.
Keep f1_row = 0.8 (coverage)This part of v2 was correct — coverage is the bottleneck. Now the base policy (SFT-v3) already produces high-coverage rollouts, so RL can reinforce them (close the pass@k→pass@1 gap).

Anti-collapse guardrails (process, not reward)

  • Monitor actor/entropy, response_length, and val has_table live; the tell-tale of the v2 collapse was response length halving + has_table→0.
  • Optional hard floor: if a rollout produces no table, clamp R to a clearly-dominated value (e.g. −0.1) so "no table" can never be a local optimum. (Kept optional — with turn-cost gone the soft has_table term already orders finish > not-finish.)
  • Short first (e.g. 10–15 steps), verify val f1 & has_table are flat-or-up before committing to a long run.

Why v3-reward ≈ "v1 + stronger coverage, minus the dead/dangerous efficiency term"

Net: it is the stable v1 reward with the row weight raised (0.5→0.8) and the (previously inert, then dangerous) turn-cost removed — applied on top of a much stronger base policy. It keeps everything that was safe, fixes what was under-powered (coverage), and deletes the exact term that caused the collapse.

6. Q&A — concepts behind the design

Notes captured while reasoning through the design (AI-engineer view).

Q: What does "inert" mean for the v1 turn_cost term?

Inert = present in the formula but always evaluated to zero, so it never affected the reward. v1 had − 0.1·turn_cost with turn_cost = min(n_turns/20, 1), but n_turns was never plumbed through from verl → it was always 0:

turn_cost = min(0/20, 1) = 0   →   −0.1 × 0 = 0   (contributes nothing)

The code looked like it rewarded efficiency, but that branch was dead. So v1's −48% turn reduction was emergent (a side-effect of chasing outcome reward), not driven by the penalty. Contrast: v2 activated it (n_turns = count("<tool_call>"), budget 12) → the penalty became real → drove the "stop early → no table" collapse. v3 removes it entirely.

Q: What is post-training, and what's its goal?

Pre-training gives a general next-token predictor (knows language + facts). Post-training reshapes that general model toward a specific behavior/objective — it redistributes probability mass toward wanted behaviors, not new facts. Here the objective is a behavioral/policy goal:

Make Qwen2.5-7B adopt the code-act batched-search pattern (enumerate all sub-queries, then loop/batch in one code block) instead of firing tool calls one at a time — saving turns/cost while keeping coverage decent. F1 is a guardrail, not the primary target.

MethodWhat it doesRole
SFT (v2→v3)Imitation — copy the distribution of good traces; moves the whole policy toward the pattern.Plants the behavior in the distribution.
GRPO (RL)Reinforcement — generate, score outcomes, push up what scored well; sharpens among behaviors already producible.Amplifies the good samples already in the distribution.

Key model: SFT plants the behavior; RL amplifies it. That's why RL-first collapsed (nothing good to amplify) and why SFT-v3 → GRPO is the right order.

Q: For GRPO, is the reward function the critical thing?

Yes — the reward defines the objective. RL optimizes what you measure, not what you want. GRPO mechanics per prompt:

1. Sample a group of G rollouts.
2. Score each with reward R_i.
3. Group-relative advantage (z-score in group):  A_i = (R_i − mean(R)) / std(R)
4. Push prob up for above-average, down for below-average, with a KL leash.

Two consequences that define the whole design:

  • If every rollout in a group scores the same → std→0 → advantage→0 → gradient→0. This is the v2 collapse: once all rollouts degenerated to "one search, no table," they scored ~equally → no signal to climb out. The reward created an absorbing state.
  • The model exploits what the reward literally says, not what you meant. Turn-cost literally said "fewer turns = more reward" → model found "1 turn, no table" satisfies it. That's reward hacking / specification gaming.

Q: Basic strategy to design/optimize the reward?

RuleWhere it shows up
a) Reward the outcome you want, not a proxy.Load-bearing terms are f1_item, f1_row; everything else is a small corrective.
b) Never let a shaping term fight the primary objective.Turn-cost penalized the very act (turns) required to finish → deleted in v3.
c) Prefer emergent over forced behavior.v1's efficiency emerged free from outcome reward; forcing it (v2) broke things.
d) Keep terms sparse & interpretable.Every extra term is another hacking surface; reward ≈ outcome − small anti-fab penalty.
e) Guarantee ordering: finishing must dominate not-finishing.Real positive has_table + optional hard floor removes the absorbing state.
f) Constrain drift with KL + the right reference.KL 0.005 to SFT-v3; the reference policy is part of the reward design.
g) Instrument for collapse; checkpoint on val, not recency.Watch entropy/length/has_table; best-by-val (v2's keep-last-2 deleted its only good ckpt).

One-sentence version: In GRPO the reward is the objective; the game is designing a reward whose only high-scoring behaviors are the ones you actually want — no degenerate shortcuts, no shaping term competing with the goal — then using KL + a strong reference + val-based checkpointing to keep RL inside that good region while it sharpens what SFT already learned.

Q: Do we need to increase has_table (0.1) to guarantee the ordering?

No — and inflating it is the wrong fix. Under v3 (R = 1.0·f1_item + 0.8·f1_row + 0.1·has_table − 0.1·constant_fill):

Rolloutf1_itemf1_rowhas_tableconstant_fillR
Give up (no table)00000
Real table, totally wrong0010+0.1
Real table, partial0.30.210+0.56

A genuine table always beats giving up by ≥ +0.1. The magnitude only needs to be strictly positive, which it is. The v2 collapse was not caused by a small has_table — it was the turn-cost paying the model to stop (finishing carried a bigger penalty than stopping). Remove turn-cost and the ordering holds regardless.

Why not inflate it: a large has_table (0.3–0.5) becomes a farmable proxy — the model emits empty/garbage tables to grab reward. You'd trade "stop early" for "spam junk tables."

Structural guarantee (belt-and-suspenders): instead of inflating the soft term, add a hard floor on no-table — cleaner and non-hackable:

if not has_table:      # produced nothing scoreable
    return -0.1        # giving up is always strictly worst

Nuance: neither the soft term nor the floor rescues you once fully collapsed (all no-table → std=0 → zero gradient). What keeps you out of that basin is starting from SFT-v3, where most rollouts already finish, so the group has variance in the right direction. Reward shaping points the gradient the right way; the good init keeps you out of the flat region.
