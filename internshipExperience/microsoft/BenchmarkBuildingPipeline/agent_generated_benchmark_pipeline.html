Building an Agent-Generated Benchmark Pipeline

Consolidated engineering experience · Microsoft internship (Web Search / Excel-Copilot research) · custom_benchmark/ — Query2Facts → Ground-Truth Agent → Log2Answer

This is a write-up of what we built, what broke, and how we fixed it while constructing a benchmark whose ground truth is produced by LLM agents that browse the live web — not by human annotators. The headline lesson: the moment both the value-producer and the value-checker are non-deterministic agents reading a changing world, "trust" stops being the currency. The currency becomes reproducible, identity-anchored evidence. Everything below is engineering fallout from that one shift.

1. What the pipeline is

Goal: turn an Excel-Copilot task (a natural-language requirement + a workbook dump) into a frozen, auditable ground-truth set we can grade any agent against. Three stages, chained so each produces a stable, versioned schema for the next:

case/ (excel_copilot_web_search_requirement.md + excel_dump.txt) │ ▼ STAGE A — Query2Facts (1 LLM call / case) Extract the atomic facts that *require the web* and are *uniquely verifiable*. Rules: R1 query asks for it · R2 not already in workbook · R3 verifiable vs an authority. Drop rules X1–X7 (unit constants, definitions, internal policy, derived values…). │ ▼ fact_spec_template.json (schema 0.6 · value=null, evidence_url=null) │ ▼ STAGE B — Ground-Truth Agent (N agent sessions / case) dispatcher → TaskBundle[] (fixed: 1/instance · discovered: 1/fact, agent enumerates) fill_loop → L3 GT-Agent fills value+url+snippet (≤2 rounds, retries nulls) → L4 Verifier independently re-checks (fetch / binary-parse / corroborate) → L6 Repair re-grounds the failures freeze → L5 fact_spec_gold.json (schema 0.7 · per instance: value+url+verified) │ ▼ STAGE C — Log2Answer (1 LLM call / gold instance) Given a graded agent's raw run log, extract what *that* agent answered for each identity, then deterministically compare to gold (numeric tol / case-fold / host suffix).
Key architectural choice: every GT-Agent and Verifier call is an isolated GitHub Copilot SDK subprocess session — no shared memory, no shared trace. The asymmetry is deliberate: the GT-Agent optimizes recall (find the value), the Verifier optimizes precision (independently re-confirm it). That isolation is what makes the gold trustworthy — and is also the source of half the obstacles below.

Obstacle 1 — Copilot SDK BYOK: 70-second silent hang Stage B

We wanted the GT-Agent / Verifier sessions to run on our team's own Azure OpenAI endpoint (BYOK) instead of the GitHub backend, to dodge rate limits. The SDK did support it via ProviderConfig — but every request hung for ~70 seconds and then returned nothing: no response, no error, no retry.

How we cornered it (elimination, then a mock server)

HypothesisTestResult
Bad creds / endpoint / api-versionPython requests.post straight to the deployment200 in <1s ✅ — ruled out
node's HTTP stack (undici)node built-in fetch replicating the CLI request200 / 2.2s ✅ — ruled out
Request body too big (58KB prompt + 17 tools)node sends 117KB bodyFast 429, not a hang — ruled out
SDK wrapper bugraw CLI -pAlso hangs → not the SDK

The breakthrough was pointing COPILOT_PROVIDER_BASE_URL at a local mock server to capture the real outbound request. The request was perfect — correct path, correct OpenAI JS client, valid chat-completions body. The only difference from our working direct calls: the CLI was reading the response as a streaming SSE chunk stream.

Root cause: the CLI defaults to streaming. Our Azure endpoint (or its front gateway) buffers the whole response instead of streaming it, so the CLI waits forever for SSE chunks that never arrive → a silent ~70s hang. A pure "client expects SSE vs endpoint doesn't stream" mismatch.
Fix: FIX turn streaming off — streaming=False in the SDK / --stream off on the CLI. Both paths immediately returned BYOK_OK. Two more papercuts fixed alongside:
  • Secondary bug: the bundled CLI returns the ping timestamp as an ISO-8601 string, but the Python SDK does int(timestamp) and crashes at start(). Monkeypatched PingResponse.from_dict to fall back to datetime.fromisoformat.
  • Quota: the 58KB system prompt 429'd instantly on a low-quota endpoint — had to route to a quota'd region (dialresearchcanadaeast).

Obstacle 2 — "GT got the value, Verifier can't reproduce it" Stage B

This was the conceptual heart of the whole project. When you build a benchmark with agents, the answer is fetched live by a non-deterministic agent from a changing world. So we kept hitting cases where the GT-Agent confidently returned a value and the independent Verifier could not get it back. Are those bugs? Sometimes — but not always.

The insight that unblocked everything: three concepts had been silently conflated:
ConceptMeansNature
correctthe value is objectively righta fact about the world
trustworthywe have reason to believe it nowa present, subjective judgment
reproduciblea second party can deterministically get it againa future, replayable guarantee
A search-summary value is often correct and trustworthy but not reproducible — it's query-dependent, re-ranked over time, with no stable anchor. A benchmark gold's currency must be reproducibility, not trust, because the gold is reused forever to judge other systems.
Common misreading we explicitly rejected: "search found it, fetch didn't → untrustworthy." Wrong. The correct statement is: "a search-only singleton that no reproducible method can independently re-derive is not gold-grade." The value may be right — but we'd rather leave an explicit gap than freeze an unreproducible singleton. The remedy is to upgrade the evidence, not discard the value: use search as a lead, then land it on a reproducible path.

Two orthogonal gates (AND)

We made the qualification rule reproducible AND identity-anchored:

  • Reproducibility answers "how do I get this value again?" — closed into 4 mechanism classes: ① direct retrieval (re-fetch / binary download+parse / headless render / API / Wayback), ② deterministic derivation (unit conversion, aggregation), ③ producer replay (frozen script + snapshot, the Tier-A/B path), ④ multi-source corroboration (N sources agree). Falls into none → UNRESOLVED.
  • Identity anchoring answers "is the value I got the one I wanted?" — entity × attribute × qualifier all match. Reproducing a value that belongs to a different entity is a false positive, so the identity gate sits on top of every reproduction method.

Obstacle 3 — Verifier blind spots → three routing strategies Stage B

A naive Verifier with one web_fetch tool produces wrong verdicts in two systematic ways. We split it into three routes:

RouteTriggerWhat it doesBlind spot it kills
① Binary parseURL ends in .xlsx/.xls/.csv/.pdflocal download → pandas/openpyxl/pypdf → inject relevant rows into the promptweb_fetch returns None on binaries; model literally can't read the table
② Standard fetchnormal HTML sourceopen the cited URL, read-only, no searching allowedroutine corroboration
③ Search corroborationprimary failed and the source is unreachable (403/404, no value read)loosen the leash, allow Bing / official mirrors to independently confirm the same valuesites that block scripts (403) but whose value is genuinely confirmable elsewhere
The critical guard: corroboration unlocks only on "source unreachable", never on "data/range mismatch." The predicate _looks_inaccessible() requires unverified + empty found_value + a 403/404-style marker. A real data error has a non-empty found_value, so it never triggers the fallback — the safety net can't mask a labeling error.

Obstacle 4 — A formal fact identity model Stage A

Early specs produced un-gradable facts: a key that mapped to several "equally valid" values, an empty key={} dirty record, and the infamous eGRID emission factor tagged reporting_year=2024 but sourced from the 2023 dataset. The fix was to formalize what a fact even is:

a fact  =  a function   f_attr : (k₁, k₂, …, kₙ) ↦ v
an instance  =  one point on that function   (key-vector → value)

Three constraints we then enforced in Query2Facts:

  • Must be a function: given the key, the value is unique. If one key yields multiple plausible values, it isn't a fact (or the key is under-specified). key={} degenerates to f()→v — empty domain, inherently unverifiable.
  • Minimal sufficient key: drop any dimension → under-determined; add any dimension → redundant. This is exactly the "identity anchor vs descriptive field" split — descriptive fields are redundant dimensions that don't determine the entity.
  • Don't forget the hidden time/version dimension: an emission factor is attr(geography, reporting_year), not attr(geography). The refresh field (static/annual/monthly…) is part of the identity model — it declares whether time enters the key and whether the value drifts.

Obstacle 5 — Log2Answer extractor: finding the answer in a 180KB distorted log Stage C

To grade a real agent we must extract, for each gold identity, what that agent actually answered — from a ~180KB mash of thoughts, Python code, stdout, 50+ Bing traces, and a final answer. Three hard sub-problems:

  1. Identity ≠ string match. "Washington, DC" appears as DC, D.C., District of Columbia; "mie" as M&IE / Meals and Incidental Expenses; fy:2026 as FY26 or even "effective Oct 1 2025 – Sep 30 2026" (no "2026" string at all). This is alias resolution + semantic equivalence — an LLM job.
  2. The same value appears 5–10×; which one was adopted? Did a later thought say "I'll use $92"? Was it ever referenced by downstream code, or just searched once and dropped?
  3. 4 layers = 4 chances to distort. agent query → Bing snippet → summarizer LLM → agent thought. The nastiest mode is self-poisoning: the agent puts a guess into the search string ("…M&IE 79 lodging June"), Bing echoes it, the summarizer reports "$79 is the rate," and the agent believes its own guess. The real gold was $92; the log contains both.
Fix direction: one LLM extract call per gold identity over a trimmed log, emitting {found, value, evidence_url, snippet, log_step_ref, confidence, reasoning}, with a stable extract_input.json v1.0 contract so the evaluator never cares which worker produced the log. The decoupling matters: any new agent is graded by wrapping its raw logs into that schema — zero evaluator changes.

Obstacle 6 — A structurally poisoned case Stage A

One retail-restock case scored 34% (10/31 verified) — the worst of all 10, and 18 of 24 unresolved errors came from it alone. Instead of blaming the agent, we ran a deterministic UPC-A check-digit audit (pure arithmetic, no network) on the workbook's 12 barcodes:

10 of 12 UPCs fail their check digit — i.e. those barcodes do not exist in reality. No manufacturer page will ever self-report a non-existent code. The single passing instance matched a real page by brand + size + case-pack, not by the fake UPC.

Diagnosis: two compounding faults. (1) The user query was partly unsatisfiable — it literally asked to "suggest a likely replacement UPC if the vendor changed it," an open research task with no self-reporting source. (2) The spec then amplified it by freezing the fake upc + internal item_id as hard identity_keys requiring the page to self-report that UPC — two fields that are structurally un-groundable on the public web.

Lesson: FIX validate the inputs before trusting the scores, and separate spec defect from unsatisfiable query. The recommendation was to drop / rebuild the case, not to tune the agent. A benchmark that fails is sometimes telling you the question is broken.

Obstacle 7 — Refreshing dynamic facts at eval time Stage B

Most facts are static (anchored to "as of some month"), but ~16 instances across 4 cases genuinely drift (weekly diesel price, daily produce wholesale, annual CDC obesity). A frozen value would be wrong by eval day. We built a portable refresh pipeline that imports zero Copilot SDK:

  • Tier A — deterministic fetchers (EIA, USDA AMS, CDC): pure urllib + regex, with a drift guard that compares the re-fetched value against the frozen one and flags anomalies. Regression: 8/9 targeted instances reproduced the frozen value at 0.00% drift.
  • Tier B — vendor catalogs (Fisher/VWR/Sigma): anti-scraping (403) → deferred to playbooks by design, not forced.
  • LLM calls go through an abstract interface so Tier A never even touches Azure credentials.

Obstacle 8 — A repair loop that never invents values Stage B

Verification failures aren't uniform, so a Layer-6 Repair-Agent triages and recovers them:

  1. Triage — rule-based classification (binary-unreadable / inaccessible → already handled, SKIP; identity-mismatch → R3; not-found → R4).
  2. Identity re-search — K independent searches anchored on the identity_keys, each re-checked by the strict identity verifier, then majority vote.
  3. Escalation — if no candidate satisfies the identity, flag UNRESOLVABLE / possible SPEC_DEFECT for a human or a targeted Query2Facts re-run. Never fabricate a value.
  4. Re-freeze with full repair_history for auditability.

Obstacle 9 — Throughput / parallelization Stage B

A 10-case batch ran 5.95 h serially; the GT-Agent is 84% of that (mean 154s, p95 385s, max 606s per call). The code spawns one Copilot subprocess per fact instance, strictly awaited in sequence, with a comment claiming concurrency "may hit auth/rate limits."

The honest finding: that comment was an untested assumption. The fix was process, not heroics: add a --max-concurrent N flag (default 1 = current behavior) and measure N=1/2/4/8 on a small case before committing a production value — rather than either blindly serializing (5.95h) or blindly firing 226 sessions at once. Theoretical ceilings: per-case ≈ 3.3×, per-fact ≈ 12–18× realistically.

3. Transferable takeaways (the interview version)

  1. Agent-built ground truth shifts the bar from "trust" to "reproducibility." If a second, isolated party can't deterministically re-derive a value with the right identity, it isn't gold — leave an explicit gap instead.
  2. Closed taxonomies beat open lists. "Forms of evidence" is endless; 4 reproduction mechanisms × an identity gate is exhaustive and gives a crisp accept/reject rule.
  3. Asymmetric producer/checker. Recall-optimized GT-Agent + precision-optimized isolated Verifier, with fallbacks that unlock only on access failures, never on data mismatches.
  4. Validate inputs before trusting outputs. A deterministic check-digit audit caught a poisoned case that would otherwise have looked like model failure. Distinguish spec defect from unsatisfiable query.
  5. Decouple via versioned schemas. fact_spec_template → fact_spec_gold → extract_input v1.0 let stages (and any future agent) evolve independently.
  6. Debug integrations with elimination + a mock server. The BYOK hang was solved by reproducing the exact request locally and noticing the streaming/SSE mismatch — not by guessing.
  7. Measure before you optimize. The "subprocesses are too heavy to parallelize" belief was never tested; the fix was a gradual, instrumented rollout.
