BenchAgents — Multi-Agent Systems for Structured Benchmark Creation

Paper: arXiv 2410.22584v2 · Butt, Chandrasekaran, Joshi, Nushi, Balachandran · University of Amsterdam + Microsoft Research (Oct 2025) · Related: evals_for_agents_notes.html · benchmark_pipeline_flowchart.html

One line. BenchAgents automates the whole benchmark-creation lifecycle — plan → generate → verify → evaluate, one LLM agent per stage, gated by human feedback — yielding diverse, hard, validated benchmarks at scale. It is the academic, synthetic-generation cousin of my custom_benchmark pipeline; the key difference is they construct ground truth by design, whereas my work solves the harder live-web ground-truth + freshness problem.

1 What they did — problem & contribution

Problem. Benchmarks bottleneck model understanding, but they saturate fast, get contaminated (train/test overlap), and are slow + expensive to hand-build. Prior auto-generation doesn't generalize: it needs custom prompt templates, domain-specific programmatic workflows, or a seed dataset to bootstrap.

Contribution. BenchAgents — a general multi-agent LLM framework that automates benchmark creation while inherently ensuring data + metric quality, with a human "developer" in the loop. It outputs (i) diverse, verified instances and (ii) executable evaluation code + metrics.

Generality is shown across language and vision via three new benchmarks:

  • BA-Calendar 2,000 instances — calendar scheduling / planning under constraints.
  • BA-Text 2,000 instances — long-form text generation under complex content constraints.
  • BA-Causal 500 instances — visual causal reasoning over 2D PyMunk physics simulations.

2 How — the 4-agent architecture

Decompose benchmark creation into 4 stages, one agent each, orchestrated by the planner, gated by human feedback. All powered by GPT-4o.

P-Agent

Planning

From a one-line NL request, decomposes into 3 sub-plans: generation params/constraints + tools; verification checks; evaluation metrics & disaggregations.

→
G-Agent

Generation code

Writes executable code. Samples structured metadata first, then fills static templates (consistency) or prompts an LLM (diversity); can call tools (PyMunk). Instance = (prompt, params, constraints).

→
V-Agent

Verification

Instance-level checks as code or prompts. Filters out anything failing any check — the filter is the quality guarantee.

→
E-Agent

Evaluation

Writes scoring code/prompts. Handles open-ended output by mixing programmatic metrics + LLM-as-judge.

Developer-in-the-loop at every stage reviews & edits plans and generated code.

V-Agent's five quality checks (reusable rubric)

CheckMeaning
ClarityInstance is understandable / unambiguous to developers and models.
CompletenessContains all parameters & constraints from the sampled metadata.
ConsistencyRealized values match the metadata (e.g. # participants in schedule = param).
FeasibilityConstraints admit ≥ 1 valid solution.
ComplexityA task-specific difficulty score (e.g. constrainedness = 1 − feasible/total slots).

3 How they proved it works

  • Checks are trustworthy — V-Agent vs. human annotations on 50 pre-filter instances agree highly (clarity 0.96–0.99, completeness 0.90–0.96). The automated filter ≈ human judgment.
  • Instances are high quality — high post-filter pass rates; BA-Causal is "correct by construction" (deterministic sim).
  • Diverse — broader parameter coverage than human-curated NaturalPlan (more participants/days + a novel buffer-time constraint).
  • Hard & solvable — accuracy drops monotonically as their complexity metric rises (valid difficulty proxy); human baseline 85% (Calendar) / 97.5% (Causal) confirms solvability.
  • Developer effort quantified (Appendix H, edit-distance pre/post feedback): Calendar needed heavy plan edits for logical constraint parsing; Causal needed little plan editing but heavy code editing for the physics sim → for tool-heavy tasks, effort shifts from planning to implementation correctness.

4 Model-analysis insights (the payoff)

Reasoning ≫ non-reasoningon scheduling & constrained writing — and the gap widens with complexity (they self-verify & re-attempt).
Negation is a universal weaknessall models struggle to exclude specified content (negative constraints).
Vision is the bottleneck, not reasoningsame causal scene as text → near-perfect; as images → collapses. Limitation is multimodal grounding.

6 What we can learn / adopt

  1. Generate from structured metadata, not free-form. Sample params/constraints first, then realize the instance — gives provable diversity & coverage. Add explicit parameter-coverage reporting to my pipeline.
  2. Make verification a first-class agent with named criteria. Borrow feasibility ("does a verifiable answer exist?") and consistency ("realized instance matches metadata?") on top of my value-correctness checks.
  3. Filtering is the quality guarantee. Over-generate + aggressively discard anything failing any check. Consider a discard-vs-repair policy (they discard; I repair).
  4. Calibrate automated checks against humans — and report the agreement number. Run a small annotator study on my Verifier to earn a credibility figure (same principle as the Anthropic eval article).
  5. A difficulty metric doubles as a coverage axis and a validity proof. Add a per-case difficulty score so the benchmark visibly discriminates between models.
  6. Quantify human effort to argue scalability. Log how much manual skill-promotion/repair editing is needed (their edit-distance trick).
  7. Design decomposing ablations. Their text-vs-image ablation isolated perception from reasoning. Analogue for me: give the agent the gold evidence page directly vs. make it search — isolating retrieval failure from extraction failure.

7 Where my work goes further

BenchAgents generates synthetic, self-contained tasks where ground truth is constructed by design — the answer is known the moment the instance is built, and there is no notion of staleness. My hard problem is the opposite: ground truth lives on the live web, so my pipeline needs web-search filling, independent URL verification, binary/JS-render handling, and a refresh/drift maintenance loop that re-derives dynamic facts at evaluation time. Their tasks are single-turn and closed; mine target agentic, open-web tasks. The refresh stage is a genuinely harder problem they explicitly avoid.
Takeaway: decomposing benchmark creation into plan → generate → verify → evaluate agents (gated by human feedback) produces validated benchmarks at scale; the trustworthiness levers are structured-metadata sampling, aggressive filtering on named quality checks, and calibrating automated checks against humans. My pipeline mirrors this skeleton — its differentiator is solving live-web ground truth + freshness.
