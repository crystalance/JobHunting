7.4 Tier A fetcher test + consolidation (automated, no human gate)

New stage custom_benchmark/refresh/consolidate.py. For every dynamic instance it (1) replays parse(frozen_snapshot) offline and (2) runs a generated fetch() that retrieves the live evidence_url and re-parses. Consolidation gate: per fact, the fetcher is promoted (deduped by source_id) only if ≥ 1 instance is BOTH offline-correct (Gate 2) AND live-ok (Gate 3) - i.e. the fetcher actually works at eval time. Everything else is marked OFFLINE_ONLY (parser correct, live fetch broken) or UNRESOLVED.

In this run every live-ok instance was also offline-correct, so the intersection equals the live set - but requiring both keeps us safe against a live fetch that returns a plausible-but-wrong value.

7.5 Gate 2 vs Gate 3 - what the two fetcher tests mean

Both gates run in consolidate.py and use the same parser, but they test different things.

Gate 2 - Offline replayGate 3 - Live fetch
Input HTMLthe frozen snapshot saved at build time (snippet in the gold)the live page fetched fresh from evidence_url now
Callparse(frozen_snapshot, key)generated fetch() → get_text(url) → parse(live_html, key)
Pass conditionoutput equals the frozen value (within tolerance)output is a non-null, plausible value (not required to equal the old value)
What it provesthe parser correctly extracts the value - producer == productthe fetcher still works end-to-end at eval time (URL reachable + page shape parseable)
Networknone (deterministic)yes (can be slow / flaky / blocked)
This run61/111 (55.0%)47/111 (42.3%)
  • Gate 2 passes, Gate 3 fails - parser is correct on saved HTML, but the live URL now 403s / redirects / changed markup. This is the 61→47 gap: the parser is right but live retrieval is broken.
  • Gate 3 deliberately does NOT require equality - the data is dynamic (a diesel price or corn future will have moved since build time), so Gate 3 only asks for a plausible value of the right kind; Gate 2 requires equality because the snapshot is frozen.
  • Consolidation is gated on Gate 2 (the trustworthy deterministic signal). Gate 3 is recorded per instance as the eval-time readiness flag - which consolidated fetchers will actually return data when the refresh agent runs them.

One line: Gate 2 = "is the parser correct?"; Gate 3 = "does the fetcher still work live?"
