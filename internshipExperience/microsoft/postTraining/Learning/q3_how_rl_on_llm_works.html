How post-training works — and what it actually does to the weights

Answer to Q3 · written for someone new to training · 2026-07-01
Your real question is the last one: "I don't have a concrete sense of how this changes every parameter in an LLM." So this doc is built bottom-up: first the one mechanism that all training shares (it's always the same three lines), then how pre-training, SFT, and RL are just different ways of choosing the loss that feeds those three lines, and finally a worked example of a single weight moving during RL.

1. There is only ONE update rule — everything else is bookkeeping

No matter what you call it — pre-training, SFT, DPO, PPO, GRPO — the model's ~8 billion numbers are changed by the exact same three steps, repeated millions of times:

1. FORWARD :  loss = L(model(input), target)      # run the model, score how "wrong" it was
2. BACKWARD:  grad = dLoss/dθ                      # for EVERY weight θ, how does loss change if θ nudges up?
3. UPDATE  :  θ  ←  θ  −  lr · grad                # nudge each weight the tiny bit that lowers loss

That is gradient descent. Step 2 (backpropagation) produces one gradient number per weight — 8 billion gradients for an 8B model. Step 3 nudges all 8 billion at once, each by lr × its own gradient (lr = learning rate, a tiny number like 1e-6).

The single most important idea: "training" never sets weights to hand-chosen values. It only ever nudges every weight a hair in the direction that reduces the loss, then re-measures, and nudges again. The only thing that differs between SFT and RL is how the loss is computed — i.e. what "wrong" means.

Analogy. Imagine 8 billion volume knobs on a giant mixing board. Backprop is a meter next to each knob saying "turn me left a smidge / right a smidge to make the song sound better." The update turns every knob by that smidge. You never know the "right" setting of any single knob — you just keep making tiny improvements. SFT and RL only disagree about what "sounds better" means.

2. What "a parameter" even is

An LLM is stacks of matrices (attention & MLP weight matrices) plus embeddings. A parameter = one number inside one of those matrices. "Qwen3-8B" means ≈ 8 billion such numbers. At inference they're frozen; at training they're the variables we optimize.

The model's output for the next token is a probability distribution over the vocabulary (~150k tokens). Every weight influences that distribution a little. Training = reshaping those distributions by moving weights.

tokens ──▶ [embed] ──▶ [attn+MLP] ×N layers ──▶ [unembed] ──▶ logits ──softmax──▶ P(next token) ▲ all the θ live here (the 8B numbers we train)

3. Pre-training vs SFT vs RL = three ways to define the loss

StageData it needsWhat "wrong" (loss) meansEffect
Pre-trainingRaw internet text (no labels)"You failed to predict the next token in this document."Learns language, facts, code syntax. Trillions of tokens.
SFT (supervised fine-tune)Labelled (prompt → ideal answer) pairs"You failed to reproduce the human-written ideal answer token-for-token."Teaches format and task behaviour by imitation.
RL (RLHF/GRPO…)Prompts + a reward function or model (no ideal answer needed)"Your own answer scored low; make high-scoring answers more likely."Teaches by trial and reward, not imitation.
"Post-training" = everything after pre-training — usually SFT then RL. Same optimizer, same three steps from §1. Only the loss differs.

This is exactly why your mentor's constraint matters: SFT's loss literally requires an "ideal answer" to copy. You don't have those (no labelled trajectories). RL's loss does not need an ideal answer — only a way to score whatever the model produced. WideSearch gives you that score for free (f1_by_row). So RL is available to you and SFT is not.

4. How SFT changes weights (the easy case first)

SFT is "predict the next token, but the document is a human demonstration." For each token position the loss is cross-entropy: −log P(correct_next_token).

target answer:  "The  capital  is  Paris ."
at position 4 the model outputs P(·):  Paris 0.10 , London 0.20 , Rome 0.05 , ...
loss_4 = −log P("Paris") = −log(0.10) = 2.30      # high loss: "Paris" wasn't likely enough
backward: which weights, if nudged, would have RAISED P("Paris")?  →  gradients
update:   nudge exactly those weights so next time P("Paris") is a bit higher (say 0.12)

Do this over millions of demonstration tokens and the weights settle where the model reliably reproduces the demonstrated style. SFT = imitation. The signal is dense (every token gives a gradient) and cheap — which is why it's stable and fast. Its limit: the model can only copy what's in the labels; it can't discover a better answer than the human wrote.

5. How RL works (general, before LLMs)

RL is learning from consequences instead of demonstrations. The loop:

┌─────────────────────────────────────────────┐ │ agent takes ACTION ──▶ environment │ │ ▲ │ │ │ │ REWARD (a number) │ │ └──── learn: make high-reward ─────────┘ │ actions more likely

Key vocabulary, mapped to plain terms:

RL termMeaning
Policy πThe decision-maker. = the LLM. Its weights θ are what we train.
ActionSomething the policy emits. For an LLM, each generated token is an action (and a whole answer is a sequence of actions).
Reward rA score for the outcome. For you: WideSearch F1. High = good.
Trajectory τOne full episode = one complete answer (all its tokens/steps).
Advantage AHow much better than average this trajectory was. The actual training signal (see §7).

The core RL idea (the "policy gradient") is one sentence:

Increase the probability of actions that led to above-average reward; decrease the probability of actions that led to below-average reward.

6. How RL-on-LLM works

Now specialize the loop to a language model. The "action" is generating tokens; the "reward" comes after the whole answer is scored.

prompt ─▶ LLM(π_θ) generates a full answer τ = (t₁, t₂, …, t_k) ← the ACTIONS │ ▼ reward function r(τ) ← e.g. WideSearch F1 = 0.72 │ ▼ "0.72 is above the group's average (0.55) → advantage +" │ ▼ nudge θ so THIS answer's tokens become MORE likely next time (and below-average answers' tokens become LESS likely)

The RL loss for an LLM

Where SFT's loss was −log P(human_token), RL's loss (policy gradient) is:

loss  =  − Advantage · log P_θ(the tokens the model itself generated)

Read it carefully — the difference from SFT is two changes:

  1. The tokens are the model's own samples, not human labels. (That's why no labelled data is needed.)
  2. Each is weighted by Advantage. If advantage is +, minimizing this loss raises log P of those tokens (do more of that). If advantage is −, it lowers them (do less of that).
That's the whole trick. RL-on-LLM reuses the identical cross-entropy machinery as SFT, but (a) trains on the model's own generations and (b) multiplies each token's gradient by a reward-derived advantage. Positive advantage ≈ "treat this as a good demonstration to imitate"; negative advantage ≈ "treat this as an anti-demonstration."

7. Watching ONE weight move during one RL step

Let's make it painfully concrete. Say weight θ = 0.4130 lives in an MLP matrix. One RL step:

Step A — generate:  policy samples an answer τ.  Its tokens include "Paris".
Step B — score:     reward r(τ) = 0.72.
Step C — baseline:  we also sampled 7 other answers; their mean reward = 0.55.
Step D — advantage: A = (0.72 − 0.55) = +0.17   (this answer was better than typical)
Step E — backward:  for weight θ, backprop asks:
                    "if θ goes up a hair, does log P(the tokens in τ) go up or down?"
                    answer (the gradient piece): ∂logP/∂θ = +0.6
Step F — combine:   RL gradient for θ  =  −A · ∂logP/∂θ  =  −(+0.17)(+0.6) = −0.102
Step G — update:    θ ← θ − lr · grad  =  0.4130 − (1e-6)(−0.102) = 0.41300010…
                                          ▲ tiny nudge UP, because this token helped a good answer

Multiply that by 8 billion weights (each gets its own ∂logP/∂θ), and by thousands of steps. No single weight is ever "set." Each just drifts by ~1e-7-scale nudges, step after step, and the aggregate of billions of such nudges is a model that produces higher-reward answers. If the answer had been bad (advantage negative), the sign of the nudge flips and that token's weights are pushed down instead.

Why RL is noisier/harder than SFT. SFT gets a gradient at every token from a known-good target — dense, low variance. RL gets one reward for a whole long answer, then smears it across all tokens via the advantage — sparse, high variance ("credit assignment": which of the 300 tokens actually deserved the 0.72?). That's why RL needs tricks: a good baseline (§8), a KL leash, many samples per prompt, tiny learning rate.

8. Putting it together: GRPO (your algorithm)

GRPO is just a smart way to compute the Advantage in §6–7 without training a second "value" network. For each prompt it samples a group of answers and uses the group's own average as the baseline:

for each prompt q:
    sample G = 8 answers  τ₁…τ₈  from the current policy
    score each:           r₁…r₈           (WideSearch F1 + shaping)
    baseline:             b = mean(r)     ← "typical" reward for this prompt
    advantage of each:    Aᵢ = (rᵢ − b) / std(r)     ← above/below average, normalized
    loss = − Σᵢ  Aᵢ · logP_θ(τᵢ)   +   β · KL(π_θ ‖ π_ref)
                                         └─ the leash from Q1: don't drift too far
    backward + update θ    (the same three steps from §1)
  • Why a group? The mean gives a free, unbiased "how good is average here?" line. Answers above it get pushed up, below it pushed down — automatically, per prompt, no critic network (saves half the GPU memory).
  • Why normalize by std? Keeps gradient scale sane whether a prompt's rewards are all near 0.9 or all near 0.1.
  • The KL term (your Q1) keeps π_θ near the frozen instruct model so it doesn't collapse — critical because, with no SFT, that reference is your only format anchor.

9. Mental model & common misconceptions

One-line summary. Post-training = repeat {run the model, score it, nudge all 8B weights a hair toward higher score}. SFT scores by "did you copy the human answer?"; RL scores by "how good was your own answer?". The mechanics of the nudge are identical.
MisconceptionReality
"RL edits specific weights for specific facts."No. Every step nudges all weights by a tiny amount; behaviour is the emergent sum of billions of nudges. No weight = one fact.
"RL needs the right answer."It needs a way to score an answer, not the answer itself. That's the whole reason it fits your no-labels situation.
"RL is a different training engine than SFT."Same optimizer, same backprop. RL just weights the token gradients by advantage and trains on self-generated text.
"Bigger learning rate = faster learning."In RL, too-big LR → collapse. RL LR (~1e-6) is ~100× smaller than SFT because the signal is noisy.
"One reward updates one action."One trajectory reward is spread across all its tokens (credit assignment) — the hard part of RL.
