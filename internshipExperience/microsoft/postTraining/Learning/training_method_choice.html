Training-method choice: why GRPO, not PPO or DPO

Decision note for the WideSearch code-act RL project · 2026-07-02
Decision. Use GRPO as the main RL method. PPO is rejected on memory + stability grounds (it needs a second critic network we can't afford on 4×A100). DPO is rejected as the main method because it throws away our scalar reward and cannot explore — though it's a valid cheap warm-up. This note explains the reasoning from our concrete constraints.

Our situation, in four facts

#ConstraintConsequence for method choice
1Verifiable reward — WideSearch f1_by_row/f1_by_item is a deterministic function of the output.We have a clean scalar score for any answer. No learned reward model needed. Favours methods that use the scalar.
2Only 1 node × 4×A100-80GB.Memory is the binding constraint. Favours methods with the fewest big networks.
3No labelled data (no gold trajectories, no preference pairs) → RL-only per mentor.Rules out plain SFT; also means DPO pairs would have to be manufactured.
4Long agentic code-act trajectories (~10 execute_python cells) that we want to improve, not imitate.Needs exploration (try new strategies) and robust credit assignment.
Read the table as a filter: (1)+(2) kill PPO's critic, (1)+(4) kill DPO's pairwise/offline nature, and all four point at GRPO — online RL, scalar reward, no critic network.

1. The three methods in one picture

All three ultimately run the same optimizer (forward → backward → nudge weights). They differ in what defines the loss and what extra machinery that requires.

2. GRPO vs PPO — the critic / value network

Both PPO and GRPO need an advantage: "was this trajectory better or worse than expected?" They differ only in how they get the "expected" baseline.

PPO: train a second network to guess the baseline

PPO runs two networks:

  • Actor = the LLM we want (the policy π_θ).
  • Critic (a.k.a. value network) = a second network, typically as large as the LLM, whose job is to predict the expected reward of a state. Advantage = actual_reward − critic_prediction.

The critic is itself trained by gradient descent. That's the cost:

What actually eats GPU memory in training is NOT just weights (~16 GB for 8B).
It's  weights + gradients + Adam optimizer states:

  Actor  (8B):  ~16 GB weights + ~16 GB grads + ~64 GB optim  ┐
  Critic (8B):  ~16 GB weights + ~16 GB grads + ~64 GB optim  ┘ ≈ TWO big models

PPO ⇒ ~2× the training-memory footprint, plus the critic is hard to train
      and unstable on long trajectories (a bad value estimate poisons every advantage).

GRPO: measure the baseline empirically — no second network

GRPO deletes the critic. Instead of predicting expected reward, it measures it: sample a group of G answers for a prompt and use the group's mean reward as the baseline.

for prompt q:
    sample G = 8 answers  τ₁…τ₈    from π_θ
    score each:           r₁…r₈     (WideSearch F1 + shaping)
    baseline  b = mean(r)           ← the "expected reward" PPO needed a whole net to guess
    advantage Aᵢ = (rᵢ − b) / std(r)
The memory payoff (the "halves memory vs PPO" claim): dropping the critic removes an ~8B-parameter model's worth of gradients + optimizer states — the single most expensive thing in training. On our 4×A100 node, where we already juggle the policy, a frozen reference model (for KL), and a vLLM rollout engine, that freed memory is often the difference between fitting and OOM.
GRPO's one cost: to get a reliable group mean you must sample G answers per prompt (e.g. 8) instead of 1 → more generation compute. But generation is cheap and parallel (vLLM); the critic's memory + instability is expensive and fragile. For a memory-limited, verifiable-reward task, trading extra rollouts for "no critic" is the right trade.

3. GRPO vs DPO

DPO (Direct Preference Optimization) learns from pairs: for one prompt, a chosen answer and a rejected answer. Its loss is essentially a classification loss — "make chosen more likely than rejected, relative to the frozen reference." No reward model, no critic, and crucially no generation during training (it runs like SFT on a fixed dataset).

Why DPO is wrong as the main method for us

  1. It throws away the magnitude of our reward. Our f1_by_row is a rich continuous number. DPO collapses everything to "chosen > rejected": 
    F1 0.88 vs 0.29   (huge, informative gap)   ┐  become the SAME
F1 0.51 vs 0.49   (basically a tie)         ┘  training signal under DPO
    We built a precise verifiable grader specifically to have that scalar — DPO discards most of it; GRPO uses it directly via the normalized advantage.
  2. It can't explore — it only imitates the pairs you hand it. DPO never generates during training, so it can only shift preference toward answers already in the dataset. It cannot discover a better code-act search strategy that isn't in any pair. Our entire goal is to improve how the model drives the loop — that needs trying new trajectories and being rewarded, i.e. online RL.
  3. Off-policy drift. DPO pairs are generated once and go stale as the policy changes; the model can't self-correct because it never samples. GRPO is always on-policy (fresh samples), which is more stable for long agentic trajectories.

4. Why "no labelled data" does not tilt us toward DPO

A natural thought: "we have no labels, and DPO is label-light, so use DPO." But we have no preference pairs either. To manufacture DPO pairs we would have to:

roll out trajectories ──▶ score each with the WideSearch grader ──▶ pair a high-F1 vs a low-F1 ──▶ DPO
Notice the trap: the moment you've built "roll out + score", you already have everything GRPO needs. DPO doesn't save you the hard part (the rollout + reward infra) — it just makes you throw away the scalar afterward and give up exploration. That's strictly less information for the same engineering cost.

5. When DPO would be the right call

  • You can't stand up online rollout infra — DPO runs like SFT (single card, very stable).
  • Your reward is inherently pairwise / subjective (human "which is better?" clicks), not a clean scalar.
  • You want a cheap, stable warm-up before committing to full RL.

6. Recommendation

Main method: GRPO. Our four constraints — verifiable scalar reward, tight GPU memory, no labels, and a task that must be improved via exploration rather than imitated — each independently point to GRPO: online RL, uses the scalar, and needs no critic network (unlike PPO) and no preference pairs (unlike DPO).

Optional pragmatic hybrid (nice de-risking for a first RL project):

Phase 0 build rollout + WideSearch reward infra (needed for GRPO anyway) Phase A (optional) roll out a few hundred trajectories, pair high-F1 vs low-F1, run DPO as a fast, stable "get the code-act format roughly right" warm-up Phase B switch to GRPO to actually optimize the scalar reward WITH exploration

DPO gives an early, low-risk win while you finish hardening the rollout infra GRPO needs — but don't stop at DPO: it leaves the reward magnitude and all exploration on the table.

One-line takeaway. PPO = GRPO + an expensive, fragile critic we can't fit. DPO = GRPO with the scalar reward deleted and exploration turned off. GRPO is the sweet spot for a verifiable-reward, memory-limited, exploration-needing agent task.
