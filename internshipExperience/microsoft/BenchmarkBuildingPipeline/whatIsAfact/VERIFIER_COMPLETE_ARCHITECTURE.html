🔑 Identity Key Verification (Core Logic)

The verifier enforces a strict identity discipline to ensure it's checking the right entity on the page.

Two-Step Identity Check

Step 1: Identity Match Validation

For each identity key in the fact spec, extract the corresponding field value from the page and compare:

FOR EACH field IN identity_keys: page_value = extract_from_source(source_content, field) instance_value = instance_key[field] IF page_value != instance_value: RETURN FAIL("identity_mismatch_{field}") IF field NOT FOUND in page: RETURN FAIL("identity_not_found:{field}")

Step 2: Anchor Confirmation

After identity keys match, verify that the matched identity anchor actually refers to what the user is asking about. Use remaining (non-identity) key fields as context.

anchor_is_consistent = check_anchor_against_context( identity_values=instance_key[identity_keys], context_fields=instance_key[non_identity_fields], user_query=task.user_query_excerpt ) IF NOT anchor_is_consistent: RETURN FAIL("identity_anchor_mismatch")

Why This Matters (Example)

⚙️ Verification Modes

The verifier supports three source-fetch modes depending on what type of data source the evidence URL points to:

Mode Source Type How It Works When Used
Standard HTML Web pages (.html, .php, etc.) Copilot fetches page HTML via web_fetch tool, scans for identity keys & values Most common; default
Binary Files .xlsx, .xls, .csv, .pdf Download file, parse locally (Excel/CSV reader or pdfplumber), inject parsed content into verifier prompt When URL points to downloadable binary document
Rendered Pages JavaScript-heavy sites (SPA, dynamic content) Use headless browser to render JS, wait for page load, then search for identity keys & values When static fetch missed value but page is renderable (opt-in via GT_RENDER_FETCH env var)

Mode Selection Logic

Detect binary?

YES → Parse Binary OR NO → Fetch HTML
↓

↓

Try Render Fallback? → If Enabled

📋 Verification Outcomes

Each verification returns a result object with three fields:

{ "verified": bool, # True only if identity match + value found "found_value": str | null, # The value found on page (or null if not found) "issues": [str] # List of reasons for failure }

Success Criteria

✅ VERIFIED (verified=true)

  • Instance has value + URL
  • URL is accessible
  • All identity keys match page values exactly
  • The claimed value is found on the page

Failure Scenarios

❌ NULL_VALUE / NO_URL

Cause: GT-Agent didn't fill the instance properly

Fix: Re-run GT-Agent with more rounds or better prompts

❌ IDENTITY_NOT_FOUND

Cause: Identity key field doesn't exist on page (e.g., "tracked_item" doesn't exist)

Fix: This is what we're fixing! Rename to web-observable field like "law_name"

❌ IDENTITY_MISMATCH

Cause: Identity key value on page doesn't match what's in the spec

Example: Spec says "state=CA" but page shows "state=TX"

Fix: Correct the identity key value in the spec to match page reality

❌ NOT_FOUND

Cause: Identity matched, but the claimed value isn't on the page

Example: Correct law ("EU AI Act") found, but "2024-06-10" date not present

Fix: Value may be missing or outdated; verify manually or try different page

⚠️ URL_INACCESSIBLE (403/404)

Cause: Cited URL is blocked or doesn't exist

Fallback: Attempts web search corroboration if enabled

Result: May still verify if corroboration finds evidence

🎯 Key Insights

1. Identity Keys Are Filters

Identity keys narrow down the "scope" of the search. They answer: "Which row/section on the page are we checking?"

Example: "Get the unemployment rate for California, Q1 2024"

  • state: "California" → identity key (scopes to CA rows)
  • quarter: "Q1" → identity key (scopes to Q1 data)
  • year: "2024" → identity key (scopes to 2024 records)

2. All Identity Keys Must Match for Verification to Pass

If ANY identity key doesn't match, verification fails immediately. This is intentional—it ensures we're checking the right scope.

3. Abstract Keys Kill Verification

"tracked_item", "period", "jurisdiction" are database/internal terms. Pages don't use them. Rename to what pages actually show: "name", "date", "location".

4. Fallback Modes Add Resilience

If the cited URL is inaccessible (403/404), the verifier can still try web search corroboration. This prevents spurious failures when pages move or block access.

5. Identity Key Simplification is the Highest-Impact Fix

68.9% of failures are identity-related. Fixing identity keys alone could improve pass rate by 5-10 percentage points (from 22.7% → 28%+). This is our validation test.
